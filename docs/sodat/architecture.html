SODAT — 農業タスク動的マッチング基盤

システムアーキテクチャ

標準化タスクを中核に、栽培状況の把握から「次に必要なタスク」を自動生成し、各画面へ届ける。作業者＝応募者は同一人材で、記録した作業がスキルとなりマッチングに還元される。

  • 栽培状況 → 次に必要なタスク
  • 実績 → スキル → マッチング反映（ループ）
  • 求人 → マッチング

「次に必要なタスク」の自動提示フロー

栽培状況の把握から、双方のダッシュボード（管理者用・栽培環境）へ自動で提示。作業者にも同じタスクが届く。

  1. 栽培状況を把握
    センサ観測・観察を cultivation_observations に追記
  2. →
  3. ② が次タスクを生成
    標準作業辞書と突き合わせ、工数・必要人数つきで算出
  4. →
  5. 双方のダッシュボードに提示
    管理者用 ＋ 栽培環境（連携）の両方に表示
  6. →
  7. 作業者にも配信
    作業者用UIに次タスクを提示。必要なら求人へ

実績がスキルになり、マッチングに効く ↺ フィードバックループ

作業者＝応募者は同一人材。作業者として記録した実績が習熟度→スキルとなり、応募者としてのマッチングを押し上げ、良いオファーが本人に還元される。

  1. 作業者として作業を記録
    実施作業を work_records に追記（TPOCast代替）
  2. →
  3. ① が習熟度→スキル認定
    作目×タスク別に集計 → worker_skills。根拠の実績が辿れる
  4. →
  5. ③ マッチングがスキル参照
    同じタスク語彙で照合 → 適合スコアが上がる
  6. ↺
  7. 応募者として良いオファー
    同一人材に還元。働くほどマッチングが強くなる好循環
凡例・補足
  • 作業者＝応募者: 同一の登録人材の2つの画面。作業者として貯めた実績が、応募者としてのスキル・マッチングに直結する。
  • 既存 は現行の栽培環境ダッシュボード。将来、管理者用ダッシュボードと連携し栽培状況を統合表示する。
  • 内部モジュール間は直接 API を持たず、すべて共通データ基盤を介して連携（API は外部境界のみ）。
  • 現段階のデータはダミー（fixtures.json）。TPOCast 連携は後日で、当面は作業者用UIが work_records を記録する。
