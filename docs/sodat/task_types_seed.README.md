# 標準作業辞書シード（task_types）

`task_types_seed.csv` は SODAT の中核マスタ「標準作業辞書」の初期シード。
R7年度報告書 §2.2.2〜2.2.3 の図2-2-1〜2-2-3（野菜／果樹／茶の標準作業仕様）を書き起こしたもの。

- 出典: `260328_R7年度報告書 統合版_r3.docx`（各図の左側表）
- 件数: 野菜23 / 果樹23 / 茶19 = **65 タスク**

## ID 規約: `base.crop`（全タスクにサフィックス）

すべての `task_type_id` は `<base>.<crop>` 形式。**例外なく作目サフィックスを付ける**。

```
teishoku.veg   定植（野菜）
sentei.fruit   剪定（果樹）
tekisai.tea    摘採（茶）
```

- `crop` ∈ `veg` | `fruit` | `tea`
- `base` = ドットの前。ID から導出可能（別変数ではない）。

### なぜ全部サフィックスか

作目で技能・適性が違うタスク（定植 vs 新植など）を取り違えないため、
**同一性は常に作目まで含めて持つ**。作目情報を ID から失わないので可逆で安全。

## 束ねはマッチング時だけ（辞書には焼き込まない）

「このタスクは作目をまたいで技能移転してよいか」は**事実ではなく仮説**。
辞書には断定させず、マッチング層で判定・検証する（4.3.2 マッチング実効性の検証）。

- 束ねキー = **ベース名**（`label_hakozume.veg` と `label_hakozume.tea` はベースが同一）
- 束ね判定の目安 = **ベース一致 かつ 適性 `◎`**（初心者軽作業は作目非依存で移転しやすい）
- 適性 `−`/`△`（専門・技能）は作目まで一致を要求

ベースが複数作目にまたがる（＝束ね候補の）タスク:
`chukou` `ekihi` `hiryo_sanpu` `josozai_sanpu` `junkai` `kansui` `noyaku_sanpu` `senbetsu` `shukaku` `tsuihi`

## 正規化の未決事項（近義語）

呼称が違うため自動では束ならないが、将来プールを検討する候補:

作目間で呼称がゆれていたため別ベースだったものを、以下に統一済み
（表示名は作目慣用を保持。ベースのみ正規化して作目横断で束ねられるようにした）:

- ~~選別 `senbetsu.veg` / 選果 `senka.fruit`~~ → **`senbetsu`**
- ~~農薬 `noyaku` / 農薬散布 `noyaku_sanpu`~~ → **`noyaku_sanpu`**（野菜・果樹・茶）
- ~~肥料 `hiryo` / 肥料散布 `hiryo_sanpu`~~ → **`hiryo_sanpu`**（野菜・果樹・茶）
- ~~除草剤 `josozai` / 除草剤散布 `jososanpu`~~ → **`josozai_sanpu`**（野菜・果樹・茶）

残る確認（ドメイン判断・急がない）:

- 作目間の 追肥・潅水・中耕・液肥・巡回・収穫 等 — 技能移転の妥当性を実証で確認
- 適性の妥当性（例: 茶「工場作業」◎、野菜の潅水3系統の粒度）

正規化する場合はベース名を一致させる（例: 選果→`senbetsu.fruit`）。
イベントログは各事実に作目コンテキストを保持し、辞書は `dictionary_version` で版管理するため、
ID の統合・分割はどちら向きも後から復旧可能。

## カラム

| 列 | 説明 |
|---|---|
| `task_type_id` | `base.crop`。全システムの結合キー |
| `base` | ベース名（ID 前半。束ねキー） |
| `task_name` | 日本語タスク名（辞書の表記） |
| `crop` | `veg` / `fruit` / `tea` |
| `process` | 工程（属性・複数値は `;` 区切り。例: 圃場巡回は複数工程に出現） |
| `spot_aptitude` | スポットワーク適性 `◎`(初心者可) `○`(指導で可) `△`(経験要) `−`(専門・スポット不可) |
| `qualification_note` | 備考（辞書の注記） |
